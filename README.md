# Контрольная работа №2

В рамках контрольной работы №2 выполнены практические задания №7-12 по дисциплине «Фронтенд и бэкенд разработка».

Работа охватывает полный цикл построения защищенного веб-приложения:

- аутентификация пользователя;
- хеширование паролей;
- JWT (access/refresh токены);
- автоматическое обновление токенов на фронтенде;
- ролевая модель авторизации (RBAC).

---

## Практические задания №7-8: основы аутентификации и JWT

### Аутентификация

Аутентификация — процесс проверки подлинности пользователя:

- идентификация (email/логин);
- верификация (проверка пароля).

### Хеширование паролей

В проекте используется `bcrypt`.
Ключевые свойства:

- односторонность (исходный пароль не восстанавливается);
- использование соли;
- замедление вычисления для защиты от брутфорса.

### JWT

После успешного входа сервер выдает JWT, который клиент отправляет в заголовке:

`Authorization: Bearer <token>`

Сервер валидирует подпись и срок действия токена.

## Практические задания №9-10: access/refresh токены и фронтенд

### Разделение токенов

- `accessToken` — короткоживущий токен для доступа к API;
- `refreshToken` — долгоживущий токен для получения новой пары токенов.

Реализована ротация refresh-токенов и endpoint обновления пары токенов.

### Хранение токенов и автоматизация

На клиенте реализовано:

- хранение токенов в `localStorage`;
- `axios` interceptors для:
  - автоподстановки `accessToken`;
  - обработки `401`;
  - запроса `refresh` и повтора исходного запроса.

### Фронтенд

Добавлен React + Tailwind интерфейс для:

- регистрации и входа;
- просмотра списка товаров;
- создания, редактирования, удаления и просмотра товара по id.

## Практические задания №11-12: RBAC (роли и права доступа)

### Аутентификация и авторизация

- **Аутентификация** отвечает на вопрос «кто пользователь?»;
- **Авторизация** отвечает на вопрос «что пользователю разрешено?».

### Роли в системе

Реализованы роли:

- `user` — просмотр товаров;
- `seller` — просмотр + создание/редактирование товаров;
- `admin` — права `seller` + управление пользователями + удаление товаров.

### Доступы на backend

Ограничение доступа реализовано через middleware:

- `authMiddleware` — проверка токена и пользователя;
- `rolesMiddleware` — проверка роли для конкретного маршрута.

Также реализованы административные маршруты `/api/users`:

- получение списка пользователей;
- получение пользователя по id;
- обновление информации пользователя;
- блокировка пользователя.

### Фронтенд с учетом ролей

В интерфейсе скрываются/показываются действия в зависимости от роли:

- пользователь видит только просмотр;
- продавец получает доступ к созданию и редактированию;
- администратор дополнительно получает управление пользователями.
